# RV1126B 视觉识别系统中文使用指南

本文档按当前代码整理，适用于 `rv1126b_vision_app` 目录下的 C++17 工程。工程目标是在 RV1126B 板端完成摄像头采集、三模型 AI 推理、坐姿/喝水逻辑、ST7789 显示、MQTT 状态发布和网页视频预览。

当前主链路已经从早期“只规划 WebRTC”调整为：

```text
V4L2 MPLANE/MMAP NV12 摄像头
  -> FramePool / LatestFrameBuffer / Drop-oldest 编码队列
  -> ImageProcessor 预处理
  -> GestureModel / PoseModel / CupModel
  -> PostureAnalyzer / DrinkDetector
  -> DisplayFace / AppState
  -> ST7789 或 LVGL 显示
  -> MQTT 可选发布
  -> MPP H.264 编码
  -> HTTP-FLV 网页预览
```

## 1. 当前代码状态

主要目录：

```text
include/        公共类型、接口和线程/队列工具
src/            主程序、设备封装、模型封装、传输封装
model/          当前随工程放置的 RKNN 模型
tools/          单图调试工具和 RTSP relay 工具
web/            HTTP-FLV 页面依赖的 flv.min.js
docs/           使用说明、模型输出说明、屏幕检查文档
third_party/    RKNN 头文件、LVGL 源码等第三方内容
```

当前默认入口是 `src/main.cpp`，主要默认值如下：

```cpp
config.camera_device = "/dev/video23";
config.frame_width = 640;
config.frame_height = 480;
config.frame_channels = 1;        // 当前真实摄像头按 NV12 处理
config.target_fps = 25;

config.enable_mock_camera = false;
config.mock_camera_frame_count = 120;

config.enable_mpp_encoder = true;
config.enable_mpp_decoder = false;
config.enable_web_stream = true;
config.enable_video_overlay = true;
config.web_stream_protocol = rv1126b::WebStreamProtocol::HttpFlv;

config.enable_mqtt = false;
config.enable_display = true;
config.enable_lvgl_display = true;
```

注意：`frame_channels=1` 不表示灰度模型输入，而是摄像头帧格式为 `PixelFormat::NV12`。模型输入会在 `ImageProcessor` 中转换为 RGB888。

## 2. 构建方式

### 2.1 CMake 开关

`CMakeLists.txt` 当前提供这些开关：

```text
RV1126B_ENABLE_RGA             默认 ON
RV1126B_ENABLE_OPENCV          默认 ON
RV1126B_ENABLE_RKNN            默认 ON
RV1126B_ENABLE_MQTT            默认 ON
RV1126B_ENABLE_MPP             默认 ON
RV1126B_ENABLE_GSTREAMER_RTSP  默认 OFF
RV1126B_ENABLE_LVGL            默认 OFF
```

这些开关打开后，CMake 仍会探测实际头文件和库。如果依赖不存在，主程序通常会退回日志或软件路径；但某些工具目标会直接链接固定库路径，实机交叉编译时需要确认 SDK 路径正确。

### 2.2 普通 fallback 构建

普通 Linux 或开发机只验证框架时，可以关闭板端依赖：

```bash
cmake -S . -B build-fallback \
  -DRV1126B_ENABLE_RGA=OFF \
  -DRV1126B_ENABLE_OPENCV=OFF \
  -DRV1126B_ENABLE_RKNN=OFF \
  -DRV1126B_ENABLE_MQTT=OFF \
  -DRV1126B_ENABLE_MPP=OFF \
  -DRV1126B_ENABLE_LVGL=OFF
cmake --build build-fallback -j
```

如果没有真实摄像头，需要在 `src/main.cpp` 中把 `config.enable_mock_camera` 改为 `true`，或把摄像头配置改成当前机器存在的 V4L2 节点。

### 2.3 RV1126B 实机构建

实机建议逐步打开 SDK 能力：

```bash
cmake -S . -B build-rv1126b \
  -DRV1126B_ENABLE_RGA=ON \
  -DRV1126B_ENABLE_OPENCV=ON \
  -DRV1126B_ENABLE_RKNN=ON \
  -DRV1126B_ENABLE_MQTT=ON \
  -DRV1126B_ENABLE_MPP=ON \
  -DRV1126B_ENABLE_LVGL=ON
cmake --build build-rv1126b -j
```

如果暂时不点屏，先关闭显示：

```cpp
config.enable_display = false;
config.enable_lvgl_display = false;
```

如果暂时没有 MPP 或网页预览，也可以关闭：

```cpp
config.enable_mpp_encoder = false;
config.enable_web_stream = false;
```

## 3. 运行方式

主程序运行：

```bash
./build-rv1126b/rv1126b_vision_app
```

常见日志：

```text
[Camera] V4L2/MPLANE/MMAP open: /dev/video23
[AI] pipeline config: three_model=true, legacy_fallback=false
[ImageProcessor] rga=..., opencv_fallback=..., geometry=...
[Web][HTTP-FLV] server list
[GestureScores] frame=...
[DisplayState] frame_id=...ening on 0.0.0.0:8080
[Display][LVGL] enabled
```

程序用 `Ctrl+C` 或 SIGTERM 退出。

### 3.1 强制 AI 运行

当前 AI 状态机默认空闲时只持续运行手势模型。识别到 Start 手势后进入 Running，才调度姿态和水杯模型；识别到 Stop 后回到 Idle。

调试姿态和水杯时，如果不想依赖 Start 手势，可以设置：

```bash
RV_FORCE_AI_RUNNING=1 ./build-rv1126b/rv1126b_vision_app
```

支持值：`1`、`true`、`on`。

### 3.2 预处理模式

当前可通过环境变量选择预处理后端和几何方式：

```bash
RV_PREPROCESS_MODE=opencv RV_PREPROCESS_GEOMETRY=resize ./rv1126b_vision_app
RV_PREPROCESS_MODE=opencv RV_PREPROCESS_GEOMETRY=letterbox ./rv1126b_vision_app
RV_PREPROCESS_MODE=rga    RV_PREPROCESS_GEOMETRY=resize ./rv1126b_vision_app
```

默认值：

```text
RV_PREPROCESS_MODE=opencv
RV_PREPROCESS_GEOMETRY=resize
```

说明：

- `resize`：直接拉伸到模型输入尺寸，当前默认。
- `letterbox`：保持比例缩放并用 114 灰边补齐，适合 YOLO 标准预处理对比。
- `rga`：主要用于直接 resize 路径。
- `letterbox` 优先走 OpenCV；OpenCV 不可用时走软件实现。

## 4. 摄像头配置

当前真实摄像头代码在 `src/CameraDevice.cpp`，使用 Linux V4L2 多平面采集：

```text
VIDIOC_QUERYCAP
VIDIOC_S_FMT: V4L2_PIX_FMT_NV12 + V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE
VIDIOC_REQBUFS
VIDIOC_QUERYBUF / mmap
VIDIOC_STREAMON
select
VIDIOC_DQBUF / VIDIOC_QBUF
```

默认节点：

```cpp
config.camera_device = "/dev/video23";
```

如果板端节点不同，先检查：

```bash
ls /dev/video*
v4l2-ctl --list-devices
v4l2-ctl -d /dev/video23 --all
```

当前代码要求驱动返回 NV12。如果 `VIDIOC_S_FMT` 后不是 NV12，会报错并退出。需要换摄像头节点或调整驱动/ISP 输出格式。

无摄像头自测时：

```cpp
config.enable_mock_camera = true;
config.mock_camera_frame_count = 120;
```

mock 摄像头输出 RGB888 彩色测试帧，主要用于线程、状态机、fallback 推理和日志链路验证。

## 5. AI 模型和逻辑

### 5.1 当前模型文件

工程当前自带：

```text
model/yolov5_gesture_rv1126b.rknn
model/yolov8n-pose-rv1126b-i8.rknn
model/yolov8n_rv1126b_i8.rknn
```

`src/main.cpp` 默认路径：

```cpp
config.gesture_model_path = "model/yolov5_gesture_rv1126b.rknn";
config.pose_model_path = "model/yolov8n-pose-rv1126b-i8.rknn";
config.cup_model_path = "model/yolov8n_rv1126b_i8.rknn";

config.use_three_model_pipeline = true;
config.use_legacy_posture_drink_model = false;
```

默认输入尺寸：

```cpp
config.gesture_input_width = 224;
config.gesture_input_height = 224;
config.pose_input_width = 640;
config.pose_input_height = 640;
config.cup_input_width = 640;
config.cup_input_height = 640;
```

### 5.2 手势模型

代码入口：`src/GestureModel.cpp`。

当前按 15 类分类输出解析。代码会对输出做 softmax，并打印 top5：

```text
[GestureScores] frame=..., top1=class_x, prob=..., top5=...
```

当前动作映射：

```text
class_0  -> Start
class_5  -> Stop
class_11 -> Heart
class_13 -> Like
其他类别 -> None
```

阈值：

```cpp
config.gesture_score_threshold = 0.60F;
```

Start 会进入 Running；Stop 会回到 Idle；Heart/Like 当前主要用于触发显示/状态更新。

### 5.3 姿态模型

代码入口：`src/PerceptionModules.cpp::PoseModel`。

当前支持 YOLOv8-pose 输出解析：

- 原始 DFL 三尺度检测输出。
- 常见 decoded tensor 输出。
- COCO 17 关键点。
- NMS 后取最高分人体。

姿态判断目前是 front45 规则：

- 使用鼻尖、耳朵、肩膀估计头前伸角和低头/后仰角。
- 头前伸阈值：约 35 度。
- 低头阈值：约 15 度。
- 后仰阈值：约 -15 度。
- 命中后映射到 `PostureState::BAD_ALERT`，否则为 `GOOD` 或 `UNKNOWN`。

关键点阈值：

```cpp
config.pose_keypoint_score_threshold = 0.35F;
```

### 5.4 水杯模型和喝水判断

代码入口：`src/PerceptionModules.cpp::CupModel` 和 `DrinkDetector`。

水杯模型按 COCO 饮水容器相关类别过滤：

```text
class_id=39 -> bottle
class_id=40 -> wine glass
class_id=41 -> cup
class_id=45 -> bowl
```

当前兼容多种 YOLOv8 检测输出：

- raw DFL 输出。
- split 输出：box / class / objectness。
- decoded tensor。
- 后处理好的扁平框输出。

当前这份 RKNN 的正常解析日志应接近：

```text
cup_yolov8_split_decoder
```

喝水判断：

- 优先用鼻尖作为头部点。
- 鼻尖不可见时，用左右耳平均点兜底。
- 计算饮水容器中心到头部点的距离，并除以人体框对角线做归一化。
- 连续命中达到配置次数后，才认为正在喝水。

相关配置：

```cpp
config.cup_score_threshold = 0.50F;
config.drink_distance_norm_threshold = 0.40F;
config.drink_consecutive_hits = 3;
```

如果没有检测到人或杯子，喝水状态回到 `NORMAL`；如果检测到人和杯子但距离超过阈值，会给出 `NEED_REMIND`。

### 5.5 旧版合并模型

代码仍保留 `PostureDrinkModel`：

```cpp
config.posture_drink_model_path = "/userdata/models/posture_drink.rknn";
config.use_legacy_posture_drink_model = false;
```

当前默认关闭。只有需要兼容旧版姿态+饮水合并模型时再打开。

## 6. 网页预览

### 6.1 当前默认协议：HTTP-FLV

当前 `src/main.cpp` 默认：

```cpp
config.web_stream_protocol = rv1126b::WebStreamProtocol::HttpFlv;
config.device_ip = "192.168.137.2";
config.web_server_port = 8080;
```

启动后访问：

```text
http://192.168.137.2:8080/
http://192.168.137.2:8080/live.flv
http://192.168.137.2:8080/state.json
```

首页由 C++ 直接生成，使用 `web/flv.min.js` 播放 `/live.flv`，并每秒轮询 `/state.json`。

也可用 ffplay 测试：

```bash
ffplay -fflags nobuffer -flags low_delay http://192.168.137.2:8080/live.flv
```

重要限制：

- HTTP-FLV 需要真实 H.264 码流。
- 当前 H.264 编码依赖 MPP。
- 如果没有编译或打开 MPP，HTTP-FLV 会丢帧并打印 `H264 encode failed`。
- 新客户端会等待 SPS/PPS 和 IDR 关键帧后才开始播放。

### 6.2 视频叠加

如果编译时找到 OpenCV，且：

```cpp
config.enable_video_overlay = true;
```

编码前会把 AI 信息绘制到画面上：

- 人体框。
- 姿态关键点和骨架。
- 水杯框。
- AI/Frame/Gesture/Posture/Drink 文本。

叠加支持 NV12、RGB888、BGR888 输入，最终再转回 NV12 送入 MPP。

### 6.3 MJPEG 模式

代码仍支持 MJPEG，但当前不是默认。需要在 `src/main.cpp` 改为：

```cpp
config.web_stream_protocol = rv1126b::WebStreamProtocol::Mjpeg;
```

MJPEG 需要 OpenCV 编码 JPEG，URL 为：

```text
http://<board_ip>:8080/stream.mjpg
```

### 6.4 WebRTC 当前状态

工程仍保留 WebRTC 适配代码：

- `H264RtpPacketizer`：Annex-B H.264 NALU 转 RTP packet，支持单 NALU 和 FU-A 分片。
- `WebRtcSignalingClient`：生成 publisher hello、candidate、AI result、RTP summary JSON。

但当前 WebRTC 还不是完整浏览器播放实现。缺少真实 DTLS/SRTP/ICE 发送层。若后续要恢复 WebRTC 方向，需要接入以下任一方案：

- libdatachannel
- WebRTC native
- GStreamer `webrtcbin`

因此当前实测网页预览优先使用 HTTP-FLV。

### 6.5 可选 RTSP relay

如果需要把 HTTP-FLV 转成 RTSP，可打开：

```bash
cmake -S . -B build-rtsp -DRV1126B_ENABLE_GSTREAMER_RTSP=ON
cmake --build build-rtsp -j
```

运行：

```bash
./build-rtsp/rv1126b_rtsp_relay \
  --input http://127.0.0.1:8080/live.flv \
  --port 8554 \
  --mount /live
```

VLC 地址：

```text
rtsp://<board_ip>:8554/live
```

## 7. MQTT

当前默认关闭：

```cpp
config.enable_mqtt = false;
```

打开后配置：

```cpp
config.mqtt_host = "127.0.0.1";
config.mqtt_port = 1883;
config.mqtt_client_id = "rv1126b-vision-node";
config.mqtt_keepalive_seconds = 30;
config.mqtt_reconnect_delay_ms = 1000;
config.mqtt_max_publish_retries = 3;
```

Topic：

```cpp
config.posture_alarm_topic = "rv1126b/alarm/posture";
config.drink_remind_topic = "rv1126b/reminder/drink";
config.result_topic = "rv1126b/vision/result";
config.app_state_topic = "rv1126b/app/state";
```

发布内容：

- 坐姿异常：`{"type":"bad_posture","frame_id":...}`
- 喝水提醒：`{"type":"drink_reminder","frame_id":...}`
- AI 摘要：`frame_id`、`bad_posture`、`drink_detected`、`box_count`
- AppState：设备模式、姿态状态、饮水状态、显示表情、手势等

如果编译时没有 `mosquitto.h` 或 `libmosquitto`，会走日志 fallback。实机部署时需要确认 CMake 输出里有：

```text
Mosquitto MQTT enabled: ...
```

QoS 建议：

- 高频 AI 结果使用 QoS 0。
- 告警/提醒可改为 QoS 1，但接收端需按 `frame_id` 或时间戳去重。
- 高频识别结果不建议 retain。

## 8. ST7789 和 LVGL 显示

默认配置：

```cpp
config.enable_display = true;
config.enable_lvgl_display = true;
config.display_tick_ms = 5;
config.display_refresh_ms = 20;
config.st7789_spi_device = "/dev/spidev1.0";
config.st7789_spi_speed_hz = 8000000;
config.st7789_width = 240;
config.st7789_height = 240;
config.st7789_rotation = 0;
config.st7789_invert_color = true;
config.st7789_gpio_dc = 128;
config.st7789_gpio_reset = 23;
config.st7789_gpio_backlight = -1;
```

当前显示后端分两种：

1. `enable_lvgl_display=true`
   - 打开 SPI 和 GPIO 后初始化 LVGL。
   - 屏幕显示 `LVGL OK`。
   - `showFace()` 的传统表情刷新会被忽略，仅打印日志。

2. `enable_lvgl_display=false`
   - 使用传统 ST7789 命令流。
   - `GESTURE_OK_FACE` 显示手绘爱心。
   - `SMILE_FACE` 显示手绘笑脸。
   - 其他表情当前主要打印日志占位。

实机点屏前请确认：

```bash
ls /dev/spidev*
gpioinfo
```

第一次建议使用 8MHz 或 16MHz，确认稳定后再提高 SPI 速度。不建议一开始直接 40MHz。

## 10. 部署检查清单

板端运行前检查：

- 摄像头节点是否存在，且支持 `V4L2_CAP_VIDEO_CAPTURE_MPLANE`。
- 摄像头是否能输出 NV12。
- `src/main.cpp` 中 `camera_device` 是否是正确节点。
- RKNN 模型文件是否在当前工作目录的 `model/` 下。
- RKNN runtime 是否能被 CMake 找到并链接。
- OpenCV 是否可用，尤其是需要视频叠加、MJPEG 或单图调试时。
- RGA 是否可用；如果不可用，是否允许 OpenCV 或软件 fallback。
- MPP 是否可用；HTTP-FLV 必须依赖真实 H.264 编码。
- `device_ip` 是否改成板子的实际 IP。
- 8080 端口是否被占用。
- `web/flv.min.js` 是否随程序工作目录一起部署。
- MQTT 是否需要打开，Broker IP 和端口是否可达。
- SPI 节点、DC/RESET/背光 GPIO 编号是否和接线一致。
- 如果启用 LVGL，CMake 是否打开 `RV1126B_ENABLE_LVGL=ON`。

## 11. 常见问题

### 11.1 程序启动失败，提示 camera open failed

优先检查：

```bash
ls /dev/video*
v4l2-ctl --list-devices
v4l2-ctl -d /dev/video23 --all
```

如果不是 `/dev/video23`，修改 `config.camera_device`。如果设备不支持 MPLANE 或 NV12，需要调整摄像头节点、ISP 配置或 `CameraDevice` 实现。

### 11.2 网页能打开，但没有视频

检查：

- 日志是否出现 `[Web][HTTP-FLV] server listening`。
- 浏览器访问的是 `http://<device_ip>:8080/`。
- `device_ip` 是否是板子真实 IP。
- MPP 是否打开成功。
- 日志是否有 `H264 encode failed`。
- 是否已经输出 SPS/PPS 和关键帧。

HTTP-FLV 当前不能使用伪码流。没有真实 MPP H.264 编码时，网页不会有正常视频。

### 11.3 姿态和水杯模型看起来没有运行

这是当前状态机的正常行为：Idle 时只跑手势模型。可以做 Start 手势，或用：

```bash
RV_FORCE_AI_RUNNING=1 ./rv1126b_vision_app
```

### 11.4 水杯框数量异常多

看日志中的 `message`：

```text
cup_yolov8_split_decoder
```

如果落到宽松解析路径，或者 `candidates`、`kept_after_nms` 非常大，说明输出布局或分数解释可能不匹配。再对照 `docs/model_outputs.md` 和 `single_image_ai_debug` 的输出排查。

### 11.5 LVGL 打开后不显示爱心表情

这是当前代码行为。`enable_lvgl_display=true` 时，传统 `showFace()` 表情会被 LVGL 后端忽略。要测试手绘爱心/笑脸，设置：

```cpp
config.enable_lvgl_display = false;
```

### 11.6 MQTT 没有消息

先确认：

```cpp
config.enable_mqtt = true;
```

再看 CMake 是否找到 Mosquitto，以及运行日志是否打印连接成功。没有 libmosquitto 时会进入日志 fallback，不会真正连 Broker。

## 12. 推荐调试顺序

1. 先关闭显示、MQTT、Web、MPP，只验证摄像头和 AI 日志。
2. 用 `RV_FORCE_AI_RUNNING=1` 跑姿态和水杯，观察 `GestureScores`、pose message、cup message。
3. 打开 OpenCV overlay，确认框和关键点坐标已经映射回原图。
4. 打开 MPP 和 HTTP-FLV，在浏览器或 ffplay 中确认视频。
5. 打开 ST7789 legacy 或 LVGL，单独确认屏幕。
6. 最后打开 MQTT，把告警和 AppState 接到业务后端。

这样可以把摄像头、模型、编码、网页、显示、MQTT 分层排查，避免多个外设同时打开时难以定位问题。
