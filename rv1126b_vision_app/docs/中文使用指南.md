# RV1126B 视觉识别系统中文使用指南

## 1. 网页端推流协议选择

网页端实时视频常见协议如下：

- MJPEG：实现最简单，浏览器兼容好，但带宽高、延迟和同步能力一般。
- WebSocket + JPEG：调试方便，适合早期验证，但视频效率不高。
- WebSocket + H.264：带宽较低，但浏览器不能直接稳定播放裸 H.264，通常需要 MSE 或 WebCodecs。
- HLS：工程成熟，适合直播分发，但延迟通常较高，不适合实时交互。
- WebRTC：浏览器原生支持实时音视频，低延迟，适合摄像头实时预览和 AI 结果同步。

本项目固定选择 WebRTC。

原因：

- RV1126B 可以用 MPP 将摄像头帧硬编码为 H.264，CPU 占用低。
- WebRTC 可以直接承接 H.264 RTP 传输，浏览器端原生解码。
- AI 识别框、坐姿状态、饮水状态可以通过 DataChannel 或业务 WebSocket 发送 JSON。
- 前端用 video 显示视频，用 canvas 叠加绘制 AI 框和 MQTT 告警信息。

推荐最终链路：

```text
RV1126B 摄像头
  -> V4L2/MMAP 采集
  -> RGA 裁剪/缩放/颜色转换
  -> RKNN 模型推理
  -> MPP H.264 硬编码
  -> WebRTC 推流到网页

AI 结果 JSON
  -> WebRTC DataChannel 或 WebSocket
  -> 前端 canvas 叠加绘制

告警/提醒事件
  -> MQTT
  -> 后端订阅 MQTT 后转发给网页
  -> 前端告警面板和 canvas 状态绘制
```

## 2. 需要修改的配置

配置集中在：

```text
src/main.cpp
```

部署时优先修改以下字段。

### 2.1 摄像头配置

```cpp
config.camera_device = "/dev/video0";
config.frame_width = 640;
config.frame_height = 480;
config.frame_channels = 3;
config.target_fps = 25;
```

如果摄像头节点不是 `/dev/video0`，用下面命令查看：

```bash
ls /dev/video*
v4l2-ctl --list-devices
```

### 2.2 模型路径

```cpp
config.gesture_model_path = "/userdata/models/gesture_control.rknn";
config.posture_drink_model_path = "/userdata/models/posture_drink.rknn";
```

后期把实际 RKNN 模型放到开发板后，修改为真实路径即可。

### 2.3 WebRTC 推流配置

```cpp
config.device_ip = "192.168.1.50";
config.web_server_ip = "192.168.1.10";
config.web_server_port = 8080;
config.webrtc_signaling_url = "ws://192.168.1.10:8080/webrtc/signaling";
config.webrtc_stun_url = "stun:stun.l.google.com:19302";
config.webrtc_video_track_id = "rv1126b-h264-video";
config.webrtc_result_channel = "ai-result-json";
config.webrtc_h264_payload_type = 96;
config.webrtc_video_ssrc = 0x1126B001U;
config.webrtc_rtp_max_payload_size = 1200;
```

只需要把 `192.168.1.50` 改成 RV1126B 开发板 IP，把 `192.168.1.10` 改成网页服务端 IP。

局域网内如果不需要公网穿透，`webrtc_stun_url` 可以保留，也可以后期改成内网 STUN/TURN 服务。

### 2.4 MQTT 配置

```cpp
config.mqtt_host = "127.0.0.1";
config.mqtt_port = 1883;
config.mqtt_client_id = "rv1126b-vision-node";
```

如果 MQTT Broker 在服务器上，例如 `192.168.1.10`，则改为：

```cpp
config.mqtt_host = "192.168.1.10";
config.mqtt_port = 1883;
```

当前 Topic：

```cpp
config.posture_alarm_topic = "rv1126b/alarm/posture";
config.drink_remind_topic = "rv1126b/reminder/drink";
config.result_topic = "rv1126b/vision/result";
```

网页端可以绘制 MQTT 信息。推荐方式是后端订阅这些 Topic，再通过 WebSocket 或 WebRTC DataChannel 转发给网页，前端统一绘制。

当前 MQTT 客户端已经提供 `libmosquitto` 真实发布路径：

```text
mosquitto_new
  -> mosquitto_connect_async
  -> mosquitto_loop_start
  -> mosquitto_publish
  -> 发布失败后有限重试和 reconnect_async
  -> mosquitto_loop_stop / mosquitto_destroy
```

如果交叉编译环境没有 `mosquitto.h` 或 `libmosquitto`，代码会自动回退到日志输出模式，用于普通开发环境自检。

QoS 建议：

- 高频 AI 结果：`QoS 0`，避免网络抖动时积压。
- 不良坐姿告警、喝水提醒：可升级为 `QoS 1`，网页端按时间戳或 `frame_id` 去重。
- `retain` 不建议用于高频识别结果，只适合设备在线状态这类最后状态消息。

交叉编译示例：

```bash
cmake -S . -B build -DRV1126B_ENABLE_MQTT=ON
```

### 2.5 MPP 与 RGA 配置

```cpp
config.enable_mpp_encoder = true;
config.enable_mpp_decoder = true;
config.input_stream_is_h264 = false;
config.video_bitrate_kbps = 2048;
config.video_gop = 25;

config.use_rga_preprocess = true;
config.fallback_to_opencv = true;
```

说明：

- `enable_mpp_encoder=true`：启用 MPP H.264 硬编码，用于 WebRTC 推流。
- `enable_mpp_decoder=true`：保留 H.264/H.265 输入硬解码入口。
- `input_stream_is_h264=false`：当前默认摄像头输出原始帧。
- `use_rga_preprocess=true`：优先使用 RGA 做裁剪、缩放、颜色转换。
- `fallback_to_opencv=true`：RGA 不可用时回退 OpenCV，保证功能可用。

## 3. 构建方式

在 RV1126B 交叉编译环境中执行：

```bash
cmake -S . -B build
cmake --build build
```

运行：

```bash
./build/rv1126b_vision_app
```

当前代码中，V4L2 摄像头采集、ST7789 Linux 基础驱动、RGA/OpenCV 图像预处理路径、RKNN 通用推理封装、Mosquitto MQTT 发布路径、MPP H.264 硬编码路径、H.264 RTP packetizer 已经落地；WebRTC DTLS/SRTP/ICE 发送层仍是工程封装。接入真实 SDK 时，替换对应类内部实现即可：

- `MppDecoder`
- `WebStreamer`
- `MqttClient`
- `DisplayDevice`
- `CameraDevice`

`MppEncoder` 当前处理链路：

```text
NV12 Frame
  -> MPP H.264 Encoder
  -> Annex-B H.264 NALU
  -> 关键帧前附带 SPS/PPS
  -> WebRTC RTP packetizer
```

如果输入是 RGB888/BGR888，代码会先软件转换为 NV12 再送入 MPP。该路径用于兜底；实际产品建议使用：

```text
V4L2/DMABUF
  -> RGA 输出 NV12
  -> MPP H.264 Encoder
```

这样可以减少 CPU 颜色转换和内存拷贝。MPP 交叉编译示例：

```bash
cmake -S . -B build -DRV1126B_ENABLE_MPP=ON
```

WebRTC 推流层当前已经实现 H.264 RTP 打包：

```text
Annex-B H.264 NALU
  -> 解析 start code
  -> 小 NALU 生成单个 RTP packet
  -> 大 NALU 使用 FU-A 分片
  -> 生成 RTP header / sequence / timestamp / marker / SSRC
```

当前 `WebRtcSignalingClient` 生成信令 JSON 和 AI 结果 JSON，并保留 outbox。后续接入方式：

```text
方案 A：libdatachannel / WebRTC native
  -> 把 RtpPacket.bytes 写入 video track
  -> AI JSON 走 DataChannel

方案 B：GStreamer webrtcbin
  -> MPP H.264 码流进入 appsrc
  -> webrtcbin 完成 RTP/DTLS/SRTP/ICE
```

注意：当前代码已经完成 RTP packetizer，但还没有实现 DTLS/SRTP/ICE 的真实网络发送。真实浏览器 WebRTC 播放必须由 libdatachannel、WebRTC native 或 GStreamer webrtcbin 承担这部分。

`ImageProcessor` 当前处理顺序：

```text
RGA 裁剪/缩放/颜色转换
  -> 失败后 OpenCV cv::resize/cvtColor
  -> 仍不可用时软件最近邻兜底
```

交叉编译时，CMake 会自动探测 `im2d.h`、`librga` 和 OpenCV：

```bash
cmake -S . -B build -DRV1126B_ENABLE_RGA=ON -DRV1126B_ENABLE_OPENCV=ON
```

RKNN 推理层当前已经提供通用封装 `RknnModel`：

```text
读取 .rknn 文件
  -> rknn_init
  -> 查询输入/输出 tensor
  -> rknn_inputs_set
  -> rknn_run
  -> rknn_outputs_get
  -> rknn_outputs_release
  -> rknn_destroy
```

默认手势模型输出格式：

```text
[None, Start, Stop, Heart]
```

默认姿态/饮水模型输出格式：

```text
[bad_posture_score, head_x, head_y, cup_score, cup_x, cup_y, cup_w, cup_h]
```

如果真实模型是 YOLO 多框输出、关键点多 tensor 输出或类别顺序不同，只需要修改：

```text
src/GestureModel.cpp::parseOutput()
src/PostureDrinkModel.cpp::parseOutput()
```

RKNN 交叉编译示例：

```bash
cmake -S . -B build -DRV1126B_ENABLE_RKNN=ON
```

## 4. 网页端绘制建议

网页结构建议：

```html
<div class="stage">
  <video id="video" autoplay playsinline muted></video>
  <canvas id="overlay"></canvas>
</div>
<div id="alarm-panel"></div>
```

绘制策略：

- WebRTC video 标签负责显示 H.264 视频。
- AI JSON 到达后，canvas 按视频宽高缩放绘制识别框。
- MQTT 告警到达后，既可以写入告警面板，也可以在 canvas 上绘制状态标签。
- AI 结果建议携带 `frame_id`，便于和视频帧做近似对齐。

## 5. 自检清单

部署前检查：

- 摄像头节点是否正确。
- RKNN 模型路径是否存在。
- MQTT Broker IP 和端口是否可达。
- WebRTC signaling 地址是否可达。
- ST7789 SPI 设备和 GPIO 编号是否正确。
- RGA 设备权限是否正确。
- MPP 编码是否能输出 H.264 关键帧和 SPS/PPS。
