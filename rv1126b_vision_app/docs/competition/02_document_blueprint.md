# 比赛文档写作蓝图

## 1. 文档总体定位

这份比赛文档应把项目定位为“基于 RV1126B 的端侧视觉健康守护系统”，而不是单模型 demo 或单一识别程序。文档主线应围绕完整工程链路展开：摄像头采集 -> 图像预处理 -> RKNN/NPU 推理 -> 多模型协同 -> 状态机判断 -> 本地显示 / 视频推流 / MQTT 同步。

写作中应突出端侧、本地推理、桌面学习/办公场景和工程完整链路。不要写成云端 AI 系统，不要使用夸张宣传语，也不要把未实测指标包装成性能优势。

## 2. 推荐作品名称

1. 基于 RV1126B 的端侧视觉健康守护系统
   - 优点：平台、端侧 AI、健康守护三要素明确，适合工程类比赛文档。
   - 缺点：桌面学习/办公场景需要在摘要和第一部分补充。
   - 是否适合正式文档：适合，推荐作为最终标题。
2. 基于端侧 AI 的桌面视觉健康守护系统
   - 优点：场景感更强，读者容易理解用途。
   - 缺点：没有直接体现 RV1126B 平台特色。
   - 是否适合正式文档：适合，可作为备选标题。
3. 面向学习与办公场景的端侧视觉健康监测系统
   - 优点：应用场景清楚，适合偏应用导向的写法。
   - 缺点：“监测”一词需要避免被理解为医疗诊断。
   - 是否适合正式文档：可用，但正文要限定为坐姿与饮水提醒。

推荐最终标题：基于 RV1126B 的端侧视觉健康守护系统。

## 3. 全文叙事主线

文档应从学生自习和桌面办公中常见的久坐、坐姿不良、饮水提醒不足等问题切入，说明作品基于 RV1126B 平台完成端侧视觉感知和本地反馈。系统通过摄像头采集桌面前方画面，使用手势识别、姿态估计和饮品检测模型进行协同推理，再由状态机完成启停控制、坐姿提醒和饮水判断。结果通过 ST7789 屏幕、HTTP-FLV/RTSP 视频链路和 MQTT 状态同步输出，体现从硬件采集、模型部署、业务判断到多端反馈的工程完整性。
