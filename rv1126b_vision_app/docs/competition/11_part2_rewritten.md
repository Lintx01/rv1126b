# 第二部分  系统组成及功能说明

## 2.1 整体介绍

本系统面向桌面学习和办公场景，采用 RV1126B 作为端侧主控平台，以摄像头作为主要视觉输入，在本地完成图像预处理、RKNN/NPU 推理、状态融合和多端反馈。系统整体可分为采集模块、预处理模块、模型推理模块、状态融合模块和输出反馈模块，各模块围绕“摄像头画面输入 -> 多模型识别 -> 业务状态判断 -> 本地与上位机反馈”的链路组织。

数据流从摄像头采集开始。摄像头向 RV1126B 输入 640×480 NV12 视频帧，预处理模块根据不同模型的输入尺寸生成对应的 RGB 图像，并记录坐标映射关系，使模型输出的人体框、关键点和杯/瓶检测框能够反算回原始画面。随后，系统分别调用手势识别、姿态估计和饮品检测三类 RKNN 模型：手势模型用于 Start、Stop、Confirm、Rock 等非接触交互；姿态模型输出人体框和 17 个关键点，用于坐姿提醒规则；饮品模型输出杯/瓶位置，用于饮水状态判断。

业务层由主控程序统一融合各类识别结果。系统处于 Idle 状态时，主要等待 Start 手势以进入运行监测；进入 Running 状态后，姿态估计、饮品检测、饮水判断和定时饮水提醒共同参与状态更新。状态融合模块将手势事件、坐姿状态、饮水状态和定时提醒结果整理为统一的系统状态，再分发到 ST7789 本地显示、HTTP-FLV 视频输出和 MQTT 状态同步链路。

输出侧保持端侧系统与上位机职责分离。ST7789 显示屏用于本地状态反馈；主程序提供 HTTP-FLV 视频输出；RTSP relay 作为独立工具，将主程序的 HTTP-FLV 转换为 VLC 可播放的 RTSP 地址；MQTT 用于同步状态、事件和遥测消息。上位机主要用于 VLC 播放、MQTT 订阅、日志查看和截图整理，不作为 AI 推理节点。系统整体不是单一模型演示，而是将采集、预处理、端侧推理、状态机判断、本地显示、视频查看和状态同步组合起来的桌面端视觉健康提醒系统。

## 2.2 硬件系统介绍

### 2.2.1 硬件整体组成

硬件系统由 RV1126B 主控平台、摄像头、ST7789 显示屏、网络连接和上位机构成。RV1126B 平台承担核心计算和控制任务，负责接收摄像头画面、运行端侧模型、执行状态融合逻辑，并驱动显示、视频和通信输出。摄像头负责获取用户上半身及桌面饮品区域画面，是手势、姿态和饮品检测的共同输入来源。ST7789 显示屏作为本地反馈端，用于呈现待机、运行、手势反馈和提醒状态。网络连接用于视频访问、RTSP relay 转换链路和 MQTT 状态同步。上位机用于查看视频、订阅消息、保存日志和整理比赛材料，不承担模型推理。

### 2.2.2 机械结构与摆放方式

当前作品采用桌面摆放式结构，适合比赛演示、功能联调和材料拍摄。摄像头应朝向用户，使视场覆盖用户头肩区域以及桌面杯/瓶区域；ST7789 显示屏应朝向用户，便于在不查看上位机的情况下观察系统状态；RV1126B 开发板、摄像头、显示屏和网络连接共同组成可演示的桌面端系统。

现阶段文档不将作品描述为已经完成外壳、CAD 结构或机械加工成果。若后续补充支架、外壳、理线结构或产品化固定方式，应以实物照片和设计资料为依据；在没有材料支撑前，仅按桌面摆放、视场覆盖和屏幕朝向说明硬件结构关系。

## 2.3 软件系统介绍

### 2.3.1 软件整体架构

软件系统采用模块化设计，由主控程序组织采集、预处理、模型推理、状态融合、显示、视频和通信链路。主控模块负责读取运行配置，初始化摄像头、图像预处理、手势模型、姿态模型、饮品模型、显示设备、视频输出和 MQTT 客户端，并在运行过程中协调各模块的数据流和状态流。

采集模块从摄像头获取 NV12 视频帧；预处理模块根据模型需求生成不同尺寸的 RGB 输入；模型推理模块分别完成手势识别、姿态估计和饮品检测；后处理与状态融合模块将模型输出转化为 Start/Stop 控制、坐姿提醒、视觉饮水判断和定时饮水提醒；输出模块将状态结果同步到 ST7789、HTTP-FLV、RTSP relay 和 MQTT。系统在 Idle 状态下以手势启动为主，在 Running 状态下运行多模型协同判断。

### 2.3.2 主控与运行状态

主控程序是软件系统的调度中心，负责把采集、AI、显示、视频和通信模块组织成统一运行链路。系统主状态包括 Idle、Running 和 Stopping，其中比赛文档重点说明 Idle 与 Running 两种业务状态即可。默认情况下系统进入 Idle，主要等待 Start 手势；识别到稳定的 Start 后进入 Running，开始进行姿态估计、饮品检测、饮水判断和提醒输出；识别到 Stop 后暂停监测并回到 Idle。

手势事件在主控层具有不同作用。Start 用于启动监测，Stop 用于停止监测，Confirm 用于确认已触发的定时饮水提醒并给出交互反馈，Rock 用于互动反馈，不改变主运行状态。为避免单帧误识别造成误触发，系统在手势结果进入业务层前加入稳定计数、触发冷却和释放锁机制。

饮水提醒由视觉判断和定时器共同参与。当前源码配置中，定时饮水提醒默认首次间隔为 30 分钟，重复提醒间隔为 5 分钟；该配置可作为“当前实现参数”写入文档，但是否达到预期提醒效果仍需通过运行日志或现场验证材料支撑。

### 2.3.3 视觉采集与图像预处理

视觉采集模块从摄像头获取 640×480 NV12 视频帧，并将图像同时提供给 AI 推理链路和视频输出链路。采集模块不直接承担模型输入转换，而是将原始帧交给预处理模块处理。

预处理模块根据模型需求生成不同输入：手势模型使用 224×224 输入，姿态模型和饮品模型使用 640×640 输入。系统在 resize 过程中记录原图尺寸、裁剪区域和模型尺寸等坐标映射信息，用于将模型输出的人体框、关键点和杯/瓶框还原到原始视频画面。该坐标反算能力是后续坐姿规则、饮水距离判断和视频叠加显示的基础。

图像处理路径以端侧平台能力为优先方向，代码支持 RGA 加速路径；在依赖或运行条件不满足时，也保留通用图像处理和软件兜底路径，确保调试阶段链路可继续运行。文档中不把 OpenCV 作为系统核心能力，而将其表述为预处理兜底和开发调试支持。

### 2.3.4 手势识别与交互控制

手势识别模块用于实现非接触式控制。系统支持 Start、Stop、Confirm 和 Rock 四类业务手势：Start 用于从 Idle 进入 Running；Stop 用于从 Running 回到 Idle；Confirm 用于确认定时饮水提醒或提供确认反馈；Rock 用于互动展示反馈。这些手势使用户无需接触按键即可完成基本交互，适合桌面学习和办公场景。

为了提升现场演示稳定性，手势结果不会直接以单帧输出触发业务动作。系统先对同一业务手势进行稳定计数，满足连续条件后才触发；触发后进入冷却时间，避免手势保持时重复触发；同时启用释放锁机制，要求手势释放或识别到其他状态后再允许下一次同类触发。通过这类状态机防抖设计，系统将模型分类结果转化为更适合实际交互的事件。

### 2.3.5 姿态估计与坐姿提醒

姿态估计模块用于输出人体框和 17 个关键点，为坐姿提醒和饮水判断提供人体位置依据。系统根据鼻尖、耳朵、肩膀等关键点计算头部姿态相关指标，并将低头、头前伸、后仰等情况映射为坐姿提醒状态。

当前规则属于工程提醒规则，而不是医学诊断规则。源码中使用关键点置信度阈值过滤无效点，并采用角度阈值判断低头、前伸和后仰等状态：头前伸角达到设定阈值时提示前伸，头部 pitch 达到正向阈值时提示低头，达到负向阈值时提示后仰。文档中保留这些规则的设计思路即可，不将其表述为医疗诊断、健康疗效或长期健康改善结论。

姿态结果还会参与系统输出：当坐姿提醒状态成立时，主控层可生成对应显示状态、视频叠加信息和 MQTT 事件。其实际提醒效果需要在第三部分通过实物显示照片、运行日志或现场截图进一步证明。

### 2.3.6 饮品检测与饮水提醒

饮品检测模块为饮水提醒提供杯/瓶位置。系统支持两类饮品模型配置：一类是 COCO class-aware 配置，可按 bottle、wine_glass、cup 等类别筛选饮品容器；另一类是 BottleBoxesOnly 配置，只输出杯/瓶候选框，不按 COCO 类别过滤。当前演示配置应以最终源码和构建日志为准；在当前主程序配置中，饮品检测采用 BottleBoxesOnly profile，其结果按无类别框处理，不应写成 `class_id=0`。

饮水状态判断融合了姿态结果和饮品检测结果。系统优先使用鼻尖作为头部点；当鼻尖不可用时，可使用左右耳关键点的平均位置作为兜底。随后计算杯/瓶中心与头部点之间的距离，并用人体框对角线进行归一化。杯/瓶靠近头部且连续命中达到配置次数时，系统判断为 DrinkDetected；有人且检测到杯/瓶但距离较远时，可进入 NeedRemind；当人体、杯/瓶或关键点信息不足时保持 Normal。

定时提醒与视觉判断共同组成饮水提醒机制。视觉链路用于判断是否出现喝水动作或是否需要提醒，定时器用于在长时间未确认饮水时触发提醒。当前配置中定时提醒首次间隔为 30 分钟，重复提醒间隔为 5 分钟；DrinkDetected 或 Confirm 可影响本次提醒状态。该模块用于桌面提醒和交互辅助，不用于声明饮水习惯改善效果或健康疗效。

### 2.3.7 本地显示、视频与通信输出

本地显示链路通过 ST7789 和 LVGL 将系统状态反馈给用户。当前代码侧设计了待机、运行、Start、Stop、Confirm、Rock、坐姿提醒、饮水提醒、饮水检测等显示状态，并通过临时反馈和基础状态回退机制控制显示节奏。ST7789 的 SPI 设备、GPIO 编号和显示效果仍需结合实物接线、运行日志和屏幕照片核对，正式成果展示应以后续照片为准。

视频输出链路由主程序提供 HTTP-FLV。摄像头帧经过轻量叠加和 H.264 编码后，由 Web 输出模块提供 `/live.flv` 视频访问地址，并可通过状态 JSON 查看当前系统状态。这里需要明确：主程序直接输出的是 HTTP-FLV，不是 RTSP。

RTSP 播放能力由独立的 RTSP relay 工具提供。该工具以主程序的 HTTP-FLV 为输入，通过 GStreamer RTSP server 转换为 `rtsp://<board_ip>:8554/live`，供上位机 VLC 播放。比赛文档应把“主程序 HTTP-FLV 输出”和“独立 RTSP relay 转换”分开描述，避免把主程序写成直接输出 RTSP。

MQTT 通信链路用于同步系统状态、事件和遥测消息。系统支持在编入 libmosquitto 且 broker 连通时进行真实网络发布；在未编入相关依赖时，代码提供日志 fallback，用于查看 topic 和 payload，但该 fallback 不等同于真实 MQTT 网络发布。正式验证 MQTT 功能时，需要以订阅窗口截图或运行日志作为证据。

### 2.3.8 单图调试工具

单图调试工具用于离线检查模型输出格式、图像预处理、坐标反算和后处理逻辑。该工具读取单张图片，复用实时链路中的预处理接口生成手势、姿态和饮品模型输入，并输出识别结果和叠加图，便于在上板联调前排查模型解析和阈值设置问题。

该工具属于研发和验证辅助，不是主程序实时运行链路，也不应在正文中喧宾夺主。正式文档只需说明其用于模型与后处理检查，具体临时输出路径和调试文件名可放在调试记录或附录中，不作为系统核心功能展开。
